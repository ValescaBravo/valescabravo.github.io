Overview

This project translates a compact set of booking requirements into a normalised relational model in PostgreSQL, makes business rules explicit via constraints, and validates the design through test and reporting queries.

What I delivered

  • PostgreSQL schema (Client, Package, Reservation).
  • ERD artefacts (clean PNG + .pgerd export).
  • Validation and reporting queries, plus exported query outputs.

BA lens

  • Requirements → business rules → constraints → tests.
  • Traceability from requirements to data entities and validation queries.
  • Clear scope, assumptions and iteration plan.

Business context & goals

A travel agency needs a structured way to store clients, purchased packages, and reservations (travel date, party size and price). The database should prevent invalid bookings and support lightweight reporting.

Goals

  • Keep client, package and reservation records consistent.
  • Prevent reservations without an existing package.
  • Enable basic reporting (volume, revenue, destinations, repeat purchasing).

Proposed KPIs (queryable)

  • Monthly reservations volume.
  • Monthly revenue (sum of reservation prices).
  • Average booking value by month.
  • Top destinations by volume and by revenue.
  • Packages per client (repeat purchasing distribution).

Design note: the current schema enforces one reservation per package via UNIQUE(PackageCode) on Reservation. This is an explicit v1 rule and is listed as a trade-off in “Next steps”.

Requirements (FR/NFR + MoSCoW)

Functional requirements

  • FR1: Store clients with a unique identifier.
  • FR2: Store packages linked to a client.
  • FR3: Store reservations linked to a package (date, party size, price).
  • FR4: Support basic operational reporting.

Non-functional requirements

  • NFR1 (Data quality): enforce PK/FK integrity and key business rules.
  • NFR2 (Maintainability): scripts versioned with clear naming.
  • NFR3 (Auditability): reproducible queries and outputs.
Must Should Could Won’t (v1)
FR1, FR2, FR3 · NFR1 FR4 · NFR2 NFR3 (outputs) · completeness snapshot Payments, refunds, auth/roles · multi-reservation per package

Data model

Entities

  • Client: customer master data.
  • Package: destination + transport + accommodation, linked to a client.
  • Reservation: booking record (travel date, party size, price), linked to a package.

Relationships (as implemented)

  • Client (1) → (0..N) Package via Package.ClientID (FK).
  • Package (1) ↔ (0..1) Reservation via Reservation.PackageCode (FK + UNIQUE).

ERD artefacts

  • Clean ERD PNG
  • ERD export (.pgerd)

Validation & outcomes

Validation focuses on integrity checks, rule enforcement and basic reporting sanity — documented in SQL and supported by exported outputs.

What was verified

  • Primary keys and foreign keys operate as expected.
  • Reservation requires a valid package (FK).
  • One reservation per package (UNIQUE on PackageCode).
  • Aggregations return expected groupings (month, destination).
